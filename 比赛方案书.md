# 工程大模型Agent 智能应用系统设计

## 赛题：OpenAnsys — 基于AI大模型的通用ANSYS APDL建模智能体系统

---

## 一、项目背景与行业痛点

### 1.1 行业背景

有限元分析（FEA）是工程设计验证的核心环节，ANSYS Mechanical APDL 是工业界广泛使用的参数化建模工具。然而，当前工程有限元建模存在以下突出痛点：

| 痛点 | 描述 |
|------|------|
| **APDL门槛极高** | ANSYS 参数化设计语言（APDL）学习曲线陡峭，工程师需要额外掌握大量命令语法和编号管理 |
| **重复劳动** | 同类结构（如建筑、机械部件、设备基础等）的建模流程高度相似，但每次仍需手动调整几何参数和网格策略 |
| **调试周期长** | APDL 命令流错误定位困难，一次建模往往需要数小时乃至数天的反复试错 |
| **知识流失** | 资深分析工程师的建模经验难以沉淀和复用，新人需要从零积累 |
| **多源数据整合** | 设计图纸、结构说明、材料参数来自多种格式（文本、图片、表格），人工整合效率低下 |

### 1.2 目标

构建一个 **基于大语言模型的通用 ANSYS APDL 智能体（Agent）系统**，能够：

1. 读取任意工程结构的建模资料（文本描述、结构化参数）
2. 自主理解结构组成、连接关系与建模意图
3. 生成 ANSYS APDL 命令流，完成从几何建模、材料定义、单元选择到网格划分的全过程
4. 自动检测 APDL 执行错误并驱动修复循环
5. 输出可供力学分析的完整有限元模型

### 2.2 核心模块说明

#### 2.2.1 主 Agent（任务规划层）

- 位于 `src/core.rs`
- 基于 Rust 实现的交互式 REPL 环境
- 负责理解用户意图、调度子模块、管理对话历史

#### 2.2.2 输入理解模块

- 位于 `src/ansys.rs` 的 `scan_input_dir()` / `build_input_summary()`
- 自动扫描 `input/` 目录，识别并分类输入资料
- 支持文本文件（`.txt`, `.md`）和工程图纸图片（`.png`, `.jpg`）
- 对多文件联合描述的工程结构进行智能整合

#### 2.2.3 APDL 生成模块

- 调用 LLM 生成初始 APDL 命令流
- Prompt 中嵌入详细的模型参数、网格策略、连接关系和边界条件
- 通过结构化 prompt 约束 LLM 生成质量
- 自动去除代码块标记，确保纯 APDL 输出

#### 2.2.4 ANSYS 执行器

- 位于 `execute_ansys()`
- 以批处理模式启动 ANSYS 19.0
- 写入临时 APDL 文件，收集 `ansys.out` 和 `.err` 输出
- 支持 `-b`（批处理）、`-dir`（工作目录）、`-j`（作业名）、`-i`（输入）、`-o`（输出）

#### 2.2.5 错误分析模块

- 位于 `classify_ansys_run_issue()` / `extract_blocking_warning_blocks()`
- 分级解析 ANSYS 错误输出：
  - **ERROR 级**：导致运行中断的严重错误
  - **堵塞性警告**：`COMMAND IS IGNORED`、`NOT PERMITTED` 等导致模型不完整的警告
  - **非堵塞警告**：如单元形状检查警告，不阻断流程

#### 2.2.6 子 Agent（APDL 修复专用）

- 位于 `RepairAgent`（`src/ansys.rs`）
- 拥有独立的上下文和历史记录
- 接受：当前 APDL + 错误报告 + 修复历史
- 输出：修复后的 APDL，不包含自然语言解释
- 支持历史压缩（token 接近上限时自动总结）
- 内置输出验证：拒绝空内容、工具转录、非 APDL 格式

### 2.3 关键流程：自动修复循环

```
LLM 生成初始 APDL
       │
       ▼
写入 draft.apdl
       │
       ▼
执行 ANSYS 批处理
       │
       ▼
读取 .err 和 .out
       │
       ├── 无错误 ──▶ 验证模型完整性（ET定义、网格生成等）
       │                    │
       │                    ├── 通过 ──▶ ✅ 建模成功
       │                    │
       │                    └── 未通过 ──▶ 进入修复
       │
       └── 有错误 ──▶ 进入修复
                        │
                        ▼
              子Agent 读取：APDL + 错误报告 + 修复历史
                        │
                        ▼
              生成修复后的 APDL
                        │
                        ▼
              重新写入 draft.apdl → 重新执行 ANSYS
                        │
                        ▼
              最多 6 次尝试，否则报告失败
```

### 3.2 数据结构化示例

输入数据结构化包含：

```

# 网格策略：不同区域采用不同网格尺寸

# 建模步骤：标准有限元建模流程
```

该数据格式可覆盖任意工程结构，仅需替换结构参数即可。

- ANSYS 19.0 APDL 命令参考（几何创建、网格划分、材料定义、单元选择）
- 通用有限元建模规范与最佳实践
- 常见 ANSYS 错误模式与修复策略
- 多种工程结构类型的建模知识（可热插拔扩展）

## 四、大模型选择与微调策略

### 4.1 基础模型选择

| 维度 | 选择 | 说明 |
|------|------|------|
| **模型** | DeepSeek / GPT 系列 | 支持多种主流大模型，通过 API 接口灵活切换 |
| **推理参数** | reasoning_effort | 支持 OpenAI、Gemini 系列模型的推理强度控制 |
| **思考能力** | enable_thinking | 支持 DeepSeek、Qwen、GLM 等模型的思考链输出 |
| **多模态** | 图片输入 | 支持将工程图纸发送给多模态模型理解 |

### 4.2 微调方向（后续规划）

当前阶段已实现的模型优化方法：

1. **Prompt Engineering**
   - 针对 ANSYS APDL 生成任务设计的专用 system prompt
   - 包含通用有限元建模的领域约束和最佳实践
   - 示例：禁止使用 `_RETURN` 占位符、要求实体共享拓扑等

2. **输出约束**
   - 自动去除代码块标记（`strip_code_fences()`）
   - 拒绝不符合 APDL 格式的回复（`validate_apdl_candidate()`）
   - 拒绝工具调用的转录内容

3. **上下文管理**
   - 历史压缩（compaction）：在 token 使用达到 80% 时自动总结历史
   - 子 Agent 独立上下文：修复 Agent 拥有独立的对话历史，不受主 Agent 上下文污染

4. **计划中的微调方案**
   - 收集成功的 APDL 生成案例和修复案例作为训练数据
   - 对基础模型进行 LoRA 微调，提升其 APDL 语法准确率
   - 构建 APDL 错误-修复对数据集，用于有监督微调

## 五、智能体系统设计

### 5.1 Agent 设计原则

1. **任务分解**：复杂建模任务拆分为"理解→生成→执行→验证→修复"的子任务
2. **子 Agent 专业化**：修复 Agent 专注于 ANSYS 错误诊断和 APDL 修复
3. **渐进式上下文**：每轮只携带必要上下文，避免 token 浪费
4. **人机协同**：支持手动审批 shell 命令，必要时人工干预

### 5.2 主 Agent 功能

| 功能 | 命令 | 说明 |
|------|------|------|
| 完整建模流程 | `/run` | 扫描输入 → APDL生成 → ANSYS执行 → 自动修复 → 模型完成 |
| 重试上一轮 | `/continue` | 从当前历史重试，不添加新消息 |
| 自动模式 | `/auto on` | 自动批准所有 shell 命令 |
| 手动模式 | `/auto off` | 每条 shell 命令需要用户确认 |
| 帮助 | `/help` | 查看所有命令 |

### 5.3 子 Agent 设计（RepairAgent）

```
RepairAgent
├── 独立上下文历史
│   ├── 独立的 HistoryFile（版本、session_id、entries）
│   └── 独立的 token 上限和压缩策略
├── 修复循环
│   ├── 接收：当前 APDL + 错误报告 + 修复笔记
│   ├── 调用 LLM 生成修复后的 APDL
│   ├── 验证输出（格式、完整性、是否含工具调用）
│   └── 失败重试（最多8次）
└── 历史管理
    ├── 自动保存修复历史到 JSON 文件
    └── token 超限时自动压缩
```

### 5.4 输出验证体系

为保障生成的 APDL 质量，系统内置多层验证：

```
LLM 输出
   │
   ▼
第1层：格式验证 ──── 去除代码块标记，检查非空
   │
   ▼
第2层：APDL 特征验证 ──── 检测 APDL 关键字（/PREP7, ET,, MP,, BLOCK, 等）
   │
   ▼
第3层：工具转录检测 ──── 拒绝包含 shell 命令、工具调用或自然语言解释的输出
   │
   ▼
第4层：ANSYS 执行验证 ──── 运行 ANSYS 并检查 .err 和 .out
   │
   ▼
第5层：模型完整性验证 ──── 检查 ET 定义、TYPE 分配、网格尺寸、网格生成命令是否存在
```

## 八、创新点

### 8.1 技术创新

1. **LLM + ANSYS 闭环自动修复**：利用 LLM 生成 APDL、ANSYS 执行验证、错误反馈驱动的自动修复循环，形成"生成→验证→修复"的自动化闭环。

2. **子 Agent 独立上下文**：修复 Agent 拥有独立的历史记录和 token 管理，避免主 Agent 的上下文污染，提高修复准确率。

3. **多层次输出验证**：从格式检查、语法特征、工具检测、执行验证到完整性验证，共五层逐级过滤，确保模型输出质量。

4. **多模型家族适配**：系统内置 8 个模型家族的参数适配，支持切换不同的底层 LLM 而无需修改业务逻辑。

### 8.2 工程创新

1. **工程结构参数结构化**：将非结构化的工程设计文档转化为结构化建模参数，降低对 LLM 理解能力的依赖，同时使系统可快速迁移至不同工程领域。

2. **渐进式建模**：采用 14 步标准建模流程，每一步都有明确的 APDL 输出目标，便于定位和修复。

3. **错误分级分类**：对 ANSYS 输出进行 ERROR / 堵塞性警告 / 非堵塞警告三级分类，避免过度修复。
