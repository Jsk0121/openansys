## Stage 2 Invocation

Stage 2 is the LS-DYNA export stage.

This stage uses the same executable:

```text
F:\ansys\19.0zhuchengxu\ANSYS Inc\v190\ansys\bin\winx64\ANSYS190.exe
```

But it adds the product mode:

```text
-p dyna
```

So the effective launch shape for Stage 2 is:

```text
ANSYS190.exe -b -dir <run_dir> -j <job_name> -i <draft.apdl> -o <ansys.out> -p dyna
```

This is the important point:

The project is **not** launching a separate standalone `ls-dyna.exe` as the main Stage 2 engine.  
Instead, it launches **ANSYS APDL in DYNA product mode**.

## Where `dyna` Comes From

The Stage 2 product string is loaded from:

1. `OPENANSYS_STAGE2_PRODUCT`

The actual shared process launcher is `execute_ansys()`.

Inside it, if a product string is present, the code appends:


## What Stage 2 Is Supposed to Do

Stage 2 is not supposed to hand-write the `.k` file as plain text.

The intended workflow is:

1. Stage 1 builds and meshes the model
2. Stage 2 resumes or reuses the Stage 1 database/model
3. Stage 2 runs ANSYS in `-p dyna` mode
4. Stage 2 uses ANSYS-side export commands such as `EDWRITE`
5. ANSYS produces the final `.k` file under `output/k/`

So the conceptual chain is:

```text
ANSYS190.exe + -p dyna + Stage 2 APDL -> ANSYS LS-DYNA-capable session -> exported .k file
```

## Practical Conclusion

For this project, the correct mental model is:

- the main executable is still `ANSYS190.exe`
- LS-DYNA capability is entered through product mode `-p dyna`
- Stage 2 is the export-oriented APDL stage
- the final `.k` file is generated by ANSYS, not by manually printing keyword text
